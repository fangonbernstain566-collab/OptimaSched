Feature: Curriculum-Based Subject/Class-Code Filtering + Timetable Allocation Progress Widget
============================================================

## Feature Summary
Schedules can now only be built from subjects that actually belong to the selected section's curriculum, instead of picking from every subject offering in the system. This is powered by a new `CurriculumSubject` model mapping program + year level to subjects, and a new `classCode` field on `SubjectOffering` (e.g. "IT101-A") that identifies the specific class section a schedule slot belongs to. The dashboard's placeholder "Timetable Allocation Status" panel was replaced with a live widget showing real allocation progress for the active school year/semester.

## Changes Made
### Frontend
- `Schedules.jsx`: Section select now cascades — choosing a Section fetches that section's curriculum subjects via `/api/subject-offerings/by-section/:sectionId`; Subject and Class Code are now two dependent dropdowns instead of one flat Subject Offering picker; added validation requiring Section, Subject, and Class Code before submit; table now shows a Class Code column.
- `ScheduleManager.jsx`: added a Subject Offering dropdown plus a read-only Class Code display field; table shows Class Code column.
- `SchedulePlotter.jsx`: tooltip and placement-confirmation dialog now display the class code alongside the subject.
- `ClassDetailsModal.jsx`: shows the subject's code under the subject name.
- `DashboardView.jsx`: swapped the static placeholder panel for the new `TimetableAllocationProgress` component.
- Added `TimetableAllocationProgress.jsx`: self-contained widget with loading skeleton, error/retry state, a progress bar, and Allocated/Remaining/Completed-Blocks stat tiles, fed by `/api/dashboard/timetable-progress`.

### Backend
- Added `subjectOffering.routes.js` — `GET /api/subject-offerings/by-section/:sectionId` resolves a section's program + year level, looks up its `CurriculumSubject` mappings, and returns subjects grouped with their nested class codes.
- `dashboard.routes.js` — added `GET /api/dashboard/timetable-progress`, computing allocation percentage from sections with at least one active schedule in the current school year + semester.
- `schedule.controller.js` — `getScheduleOptions` now also returns `subjectOfferings` (with subject included) in dropdown data.
- `app.js` — mounted the new subject-offering routes at `/api/subject-offerings`.

### Database
- `Section`: added `program` (String) and `yearLevel` (Int), backfilled from the existing `"<PROGRAM>-<YEAR_LETTER><BLOCK>"` naming convention.
- `SubjectOffering`: added a required, unique `classCode` field; existing rows backfilled as `"<SubjectCode>-<Letter>"`.

### Bug Fixes

4. On the dashboard, `TimetableAllocationProgress` calls `GET /api/dashboard/timetable-progress` on mount, which counts total sections vs. sections with at least one non-deleted schedule in the current school year + semester, and renders a progress bar and stat tiles.

### Backend
- New `timmy.service.js` — `recommendSlots(scheduleId)`: candidate generation, hard-constraint filtering, scoring, top-3 selection with `capacityMatch`, `equipmentMatch`, `conflictStatus`, `availability`, and `rank` fields.
- New `timmy.controller.js` / `timmy.routes.js` — `POST /api/timmy/recommend`, gated to ADMINISTRATOR/REGISTRAR_SCHEDULER.

Extended the existing Admin/Registrar "Recently Deleted" (soft-delete + restore + permanent-delete) pattern to Cashier (Payments) and Instructor (cancelled Schedule Requests) — both previously had no delete capability at all on their own data.

## Changes Made
### Frontend
- `CashierDashboard.jsx` — added a Delete action to the transactions table.
- New `CashierRecentlyDeleted.jsx` — restore/permanently-delete soft-deleted payments.
- `InstructorSchedules.jsx` cancel action now soft-deletes instead of hard-deleting (backend change only, same UI flow).
- New `InstructorRecentlyDeleted.jsx` — restore a cancelled request back to PENDING, or permanently delete it.
- "Recently Deleted" sidebar group now shows different children per role (Deleted Payments for Cashier, Deleted Requests for Instructor, existing entries for Admin/Registrar).

### Backend
- `payment.service.js`/`controller.js`/`routes.js` — added soft-delete/restore/permanent-delete/list-deleted; `list()` and `getStats()` now filter `isDeleted: false` by default.
- `scheduleRequest.routes.js` — cancel (`DELETE /:id`) now soft-deletes; added `GET /mine/recently-deleted`, `PATCH /:id/restore`, `DELETE /:id/permanent`; active-list queries now filter `isDeleted: false`.

### Database
- Added `Payment.isDeleted` (Boolean, default false), `Payment.deletedAt` (DateTime?)
- Added `ScheduleRequest.isDeleted` (Boolean, default false), `ScheduleRequest.deletedAt` (DateTime?)

### API
- New: `DELETE /api/payments/:id`, `PATCH /api/payments/:id/restore`, `DELETE /api/payments/:id/permanent`, `GET /api/payments/recently-deleted`
- New: `GET /api/schedule-requests/mine/recently-deleted`, `PATCH /api/schedule-requests/:id/restore`, `DELETE /api/schedule-requests/:id/permanent`

### Validation
- Restore/permanent-delete both verify ownership (instructor can only restore their own requests) and correct current state (must be soft-deleted first).

### Bug Fixes
- None.

## Technical Flow
1. Cashier deletes a payment → soft-deleted, disappears from dashboard/stats, appears in Recently Deleted → Restore brings it back into active stats, or Permanently Delete removes it for good.
2. Instructor cancels a request → soft-deleted (was previously a hard delete) → appears in their Recently Deleted → Restore sets it back to PENDING status.

Feature: In-App Notification System
============================================================

## Feature Summary
Added a real notification bell with a `Notification` model backing it — not just UI. Wired into the Schedule Request lifecycle: instructor submits a request → all Admin/Registrar accounts get notified; Admin/Registrar approves/rejects → the requesting instructor gets notified, including the reviewer's reason on rejection.

## Changes Made
### Frontend
- New `NotificationBell.jsx` — badge-counted bell in the header, polls unread count every 30s, popover lists recent notifications, click marks read + navigates to the relevant page, "Mark all read" action.
- `DashboardLayout.jsx` — bell added to the AppBar next to the user avatar.
- New `notificationApi.js` service wrapper.

### Backend
- New `notification.service.js` — single/fan-out create (by user IDs or by role names), list, unread count, mark-read, mark-all-read.
- New `notification.routes.js` — mounted at `/api/notifications`, available to any authenticated role (not role-restricted, since notifications are per-user).
- `scheduleRequest.routes.js` — notification triggers added on submit (fan-out to ADMINISTRATOR + REGISTRAR_SCHEDULER), approve, and reject (both to the requesting instructor).

### Database
- Added `Notification` model (userId, title, message, type, link, isRead, createdAt).

### API
- New: `GET /api/notifications/mine`, `GET /api/notifications/unread-count`, `PATCH /api/notifications/:id/read`, `PATCH /api/notifications/read-all`

### Validation
- `markRead` verifies the notification belongs to the requesting user before updating.

### Bug Fixes
- None.

## Technical Flow
1. Instructor submits a schedule request → `NotificationService.createForRoles(['ADMINISTRATOR','REGISTRAR_SCHEDULER'], ...)` fans out a notification to every Admin/Registrar account.
2. Admin/Registrar approves or rejects → `NotificationService.create(instructorUserId, ...)` notifies the instructor, with the reviewer's `reviewNotes` included in the message on rejection.
3. `NotificationBell` polls `/api/notifications/unread-count` every 30s and fetches the full list when opened; clicking an item marks it read and navigates via its `link`.
